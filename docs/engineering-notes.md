# 工程思维记录

## 1. 为什么先定义 Message

多 Agent 系统中，模块之间如果随意传递字符串，会很快失控。统一 `Message` 可以让车端、云端、日志、测试共享同一套协议。

面试表达：

> 我先抽象统一消息协议，是为了让多 Agent 之间通过稳定的数据结构协作，而不是依赖临时字符串拼接。后续接入真实端云通信、链路追踪或日志系统时，可以复用这套协议。

## 2. 为什么 SafetyAgent 前置

车载场景中，危险指令不能依赖云端判断，也不能进入执行链路。安全拦截必须在本地优先完成。

面试表达：

> 车载 AI 应用和普通聊天应用不同，模型回答是否聪明不是唯一目标。涉及动力、制动、转向、加速的指令必须先被本地安全策略拦截。

## 3. 为什么第一版不用真实向量库

第一版重点是验证 RAG 在架构中的位置。使用字典和列表模拟检索，可以保持项目轻量、可复现，并保留后续替换向量库的接口边界。

面试表达：

> 第一版没有直接引入向量数据库，是因为我希望先验证端云协同主链路。当前 RAG 是用本地知识库模拟的，后续可以把底层检索替换成 embedding 和向量库，上层 Agent 不需要大改。

## 4. 为什么拆成车端和云端 Agent

车端关注实时性、安全性和断网可用，云端关注复杂决策、用户画像和外部生态聚合。两者职责不同，因此拆成不同目录和模块。

面试表达：

> 我把车端和云端分开，是因为它们的约束不同。车端必须稳定、快速、可离线，云端可以做更复杂的规划和个性化决策。

## 5. 为什么使用 unittest

项目目标是 offline、轻依赖、无冲突，所以第一版使用 Python 标准库 `unittest`。这保证任何 Python 3.8~3.11 环境都能直接运行测试。

面试表达：

> 我没有为了测试引入额外依赖，而是用标准库先覆盖核心链路。这符合 offline 项目的可复现要求。

## 6. 当前测试覆盖重点

当前测试覆盖：

- 统一消息协议。
- 内置知识库和车辆状态数据。
- 危险指令拦截。
- 本地意图识别。
- 云端 Agent 调度。
- 在线云端执行。
- 离线本地兜底。

这些测试覆盖的是系统最容易出错、也最值得面试展示的核心路径。
