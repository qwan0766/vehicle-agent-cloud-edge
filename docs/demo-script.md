# 面试演示脚本

## 1. 30 秒开场

这是一个车载智能座舱端云协同 Multi-Agent 原型系统。它不是普通聊天机器人，而是覆盖了意图识别、安全拦截、端云路由、用户画像召回、路线规划、断网兜底、网页可视化和数据闭环的 AI 应用工程项目。

## 2. 3 分钟演示顺序

网页已经内置 **面试演示模式** 面板。面试时可以直接按面板里的步骤点击，页面会自动填入指令、切换网络状态并运行，同时展示每一步的讲解重点。

### 第一步：在线导航

操作：

```text
用户：user_001
网络：ONLINE
指令：导航去蔚来中心
```

讲法：

> 这条链路会先经过 LocalIntentAgent 做本地意图识别，再经过 SafetyAgent 做安全判断。网络在线时进入云端 CloudScheduleAgent，调度用户画像、外部生态和路线规划 Agent。因为 user_001 有路线偏好高速，所以最终路线结果会体现“结合用户路线偏好高速”。

### 第二步：离线车控

操作：

```text
网络：OFFLINE
指令：打开座椅加热
```

讲法：

> 断网时系统不会访问云端，而是走车端本地兜底。这里由 CarControlAgent 执行座椅加热，同时数据闭环记录一次座椅加热偏好。

### 第三步：危险指令

操作：

```text
指令：加速到100km/h
```

讲法：

> 这条指令语义上属于 CAR_CONTROL，但 SafetyAgent 和 SafetyPolicy 会判断为危险，不会进入执行链路。车载 AI 应用里，安全策略优先级高于模型能力。

### 第四步：多用户切换

操作：

```text
切换 user_002
指令：电量低
```

讲法：

> 切换用户后，RAG 召回区域会出现 user_002 的画像，例如充电提醒 20%。这说明系统不是写死单用户，而是支持用户上下文隔离。

### 第五步：数据闭环

观察：

```text
数据闭环：RECORDED
偏好更新：路线偏好高速 +1 / 座椅加热偏好 +1
```

讲法：

> 每次执行都会生成 UsageEvent，并基于规则生成 PreferenceUpdate。当前用 JSONL 模拟，后续可以迁移到数据库、埋点平台或画像服务。

## 2.1 面试演示模式步骤

当前内置 5 个步骤：

| 步骤 | 指令 | 展示重点 |
| --- | --- | --- |
| 在线导航端云协同 | `导航去蔚来中心` | 端云协同、RAG、真实地图、DeepSeek |
| 车控指令不误调地图 | `温度调到24度` | 车控分流，不调用 `route.plan` |
| 低电量补能规划 | `电量低` | 补能 RAG、高德 POI、路线规划 |
| 危险指令安全拦截 | `关闭AEB` | SafetyAgent / SafetyPolicy 拦截 |
| 真实接口失败解释 | `导航去巴黎` | 在线失败不造假，展示友好错误解释 |

推荐讲法：

> 我给页面加了面试演示模式，不是为了多一个按钮，而是把项目的核心能力串成故事线。每一步都对应一个工程点：端云协同、按意图分流、补能规划、安全拦截和真实 API 失败处理。

## 3. 5 分钟扩展讲法

如果面试官愿意深入，可以继续讲：

- 为什么先做 offline：保证可运行、可复现、无依赖冲突。
- 为什么用 Retriever 抽象 RAG：后续可替换 BM25、embedding、FAISS 或 Milvus。
- 为什么安全策略独立：防止危险指令绕过业务流程。
- 为什么数据闭环独立成 feedback 包：避免日志逻辑污染 Agent。
- 为什么网页通过 app_model 适配：前端只依赖 JSON 协议，不耦合 Agent 内部实现。

## 4. 常见追问回答

### 为什么不用真实大模型？

第一版目标是验证端云协同、Agent 边界、安全拦截和数据闭环。真实 LLM 可以作为后续增强，但不应该绕过确定性的安全策略。

### 如果要接真实向量库怎么做？

保留 `retriever.search(query, top_k)` 接口，把 `SimpleRetriever` 替换为 FAISS 或 Milvus 实现，上层 Agent 不需要重写。

### 数据闭环真的会影响决策吗？

当前已经有 `PreferenceStore` 累计偏好状态。后续可以让动态偏好权重进入用户画像召回和路线规划排序。
