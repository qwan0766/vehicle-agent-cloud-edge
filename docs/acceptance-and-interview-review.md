# 验收与面试复盘

## 1. 本轮目标

本轮不是继续堆功能，而是把项目推进到“可验收、可复盘、可面试讲清楚”的状态。

新增交付：

- 一键验收脚本：`scripts/run_acceptance.py`
- 验收脚本测试：`tests/test_acceptance_runner.py`
- 验收报告输出：`reports/acceptance_report.md`
- 面试复盘文档：本文档

## 2. 为什么要做验收脚本

AI 应用 demo 很容易陷入“页面看起来能跑”的状态，但面试官真正关心的是：

- 这个系统怎么证明自己没有退化？
- 外部 API、LLM、RAG、安全拦截是否都有可重复验证方式？
- 真实接口失败时，系统是崩溃、瞎编，还是能定位问题？
- 多 Agent 调用链是否可观测？

所以我把验收拆成四层：

1. 单元测试：验证模块级行为。
2. 离线评测：验证意图、安全、状态、RAG 命中。
3. Provider smoke test：验证 DeepSeek、高德、天气、充电站等外部能力。
4. 在线代表输入矩阵：验证真实端云链路和关键边界样例。

## 3. 一键验收命令

```bash
python scripts/run_acceptance.py
```

执行后会生成：

```text
reports/acceptance_report.md
```

可选参数：

```bash
python scripts/run_acceptance.py --skip-provider-smoke
python scripts/run_acceptance.py --skip-online-matrix
python scripts/run_acceptance.py --report-path reports/custom_report.md
```

网页端也会读取最近一次验收结果：

- 后端接口：`GET /api/acceptance`
- 首页模块：`验收报告`
- 展示内容：总体状态、生成时间、unit tests / offline evaluation / provider smoke / online matrix 四个步骤

面试表达：

> 我把验收报告接入了 Web 展示，但不会在打开页面时重新跑真实 API。页面展示的是最近一次可复现验收的结果，这样既直观，也避免了演示页面初始化时被外部接口稳定性影响。

## 4. 这轮测试覆盖了什么

### 4.1 意图识别覆盖

覆盖输入包括：

- `导航去蔚来中心`
- `到外滩`
- `导航去 121.50,31.25`
- `温度调到24度`
- `打开座椅加热`
- `电量低`
- `我的偏好`
- `打开视频网站`

关键修复：

- 短导航表达 `到外滩` 需要识别为 `NAVIGATION`。
- 显式 GPS `121.50,31.25` 需要作为目的地，不再走地理编码。
- `打开视频网站` 不能因为包含“打开”就误判成车控。

面试表达：

> 我没有只测 happy path，而是主动设计了容易误判的输入，比如“打开视频网站”和“到外滩”。这些输入能逼出意图识别的边界问题。

### 4.2 安全拦截覆盖

覆盖输入包括：

- `加速到100km/h`
- `关闭AEB`
- `接管方向盘`
- `踩刹车`

关键修复：

- 危险词扩展到 AEB、方向盘、接管等辅助驾驶相关表达。
- 危险判断大小写不敏感。
- 危险指令仍然可以识别其意图类型，但执行状态必须是 `BLOCKED`。

面试表达：

> 在车载场景里，安全不是靠 LLM 自觉，而是必须前置成确定性策略。我的设计是先识别意图，再由 SafetyAgent 和 SafetyPolicy 判断能否执行。

### 4.3 地图与目的地覆盖

覆盖输入包括：

- `到外滩`
- `导航去萧山机场`
- `导航去 121.50,31.25`
- `导航去巴黎`

关键修复：

- 高德地理编码对短地名会有城市歧义，例如“外滩”可能解析到错误城市。
- 新增城市语境推断：`外滩` 约束到上海，`萧山机场` 约束到杭州。
- 对 `巴黎` 这类高德驾车路线演示无法覆盖的目的地，不做离线兜底，而是返回友好错误解释。

面试表达：

> 真实 API 接入后，问题不再是“接口能不能调通”，而是“接口返回是否符合业务语境”。外滩这个问题说明我开始处理真实工程里的数据歧义。

### 4.4 云端 Agent 编排覆盖

本轮发现一个关键编排问题：

```text
温度调到24度
```

旧逻辑会进入云端后统一调用 `route.plan`，导致车控指令也被拿去解析目的地。

修复后：

- `NAVIGATION` / `CHARGE_PLAN`：调用 `route.plan`、`provider.geocode`、`provider.map.route`
- `CAR_CONTROL`：只调用画像、生态和 `decision.summarize`
- `PERSONALIZE`：只调用画像、生态和 `decision.summarize`
- `UNKNOWN` / `DANGEROUS`：不进入云端工具链

面试表达：

> Multi-Agent 不是 Agent 越多越好，而是要按任务类型调度。车控指令调用路线 Agent 是典型的编排错误，所以我把 CloudScheduleAgent 改成按 CommandType 分流。

## 5. 当前验收矩阵

`scripts/run_acceptance.py` 的在线代表输入包括：

| 输入 | 期望 |
| --- | --- |
| `到外滩` | 导航成功，调用高德地理编码和路线规划 |
| `导航去 121.50,31.25` | 导航成功，目的地来源为显式 GPS |
| `温度调到24度` | 车控成功，不调用路线 Agent |
| `我的偏好` | 个性化成功，不调用路线 Agent |
| `AEB是什么` | 信息查询成功，不调用路线 Agent，不执行车控动作 |
| `导航去北京` | 返回 `NEEDS_CLARIFICATION`，要求补充唯一目的地 |
| `打开视频网站` | 未知指令，被策略层拦截 |
| `关闭AEB` | 危险车控，被安全层拦截 |
| `电量低` | 补能规划成功，调用路线规划 |

## 6. 面试时可以怎么讲

### 6.1 项目一句话

> 我做的是一个车载 Multi-Agent 端云协同 AI 应用原型，核心是把车端安全、云端 LLM 决策、RAG 检索、真实地图/天气/充电站 API 和网页可视化串成一条可观测、可测试的工程链路。

### 6.2 工程亮点

- 端云三层架构清晰：车载执行层、端云通信层、云端决策层。
- 8 个 Agent 职责明确，不把所有逻辑塞进一个大函数。
- LLM 只接在需要语义生成和决策的位置，安全和车控保持确定性。
- 真实 API 接入不是简单调用，而是纳入 Provider、trace、错误解释和验收体系。
- 有离线评测和一键验收脚本，能证明项目持续可用。

### 6.3 工程硬化后的面试讲法

这轮优化的重点不是再接一个 API，而是补齐工程语义边界：`INFO_QUERY` 让安全知识问答从未知指令中拆出来；`NEEDS_CLARIFICATION` 让模糊目的地成为正常对话状态；目的地候选契约让低置信度地图结果不能直接启动导航。这些变化体现的是业务状态建模、模块边界和安全优先的工程思维。

### 6.4 可以主动讲的 bug

面试时可以主动讲这几个问题，因为它们很像真实工程：

1. `打开视频网站` 被误判为车控。
   - 修复方式：收紧车控领域词，不让通用动词直接决定意图。

2. `到外滩` 被高德解析到错误城市。
   - 修复方式：为短地名增加城市语境推断。

3. `温度调到24度` 被云端调度到路线 Agent。
   - 修复方式：CloudScheduleAgent 按 `CommandType` 分流。

4. `关闭AEB` 没有被危险拦截。
   - 修复方式：扩展安全关键词并加入测试矩阵。

5. `导航去北京` 这类模糊地点不应直接规划路线。
   - 修复方式：增加 `NEEDS_CLARIFICATION`，要求用户补充唯一目的地。

6. 地图返回低置信度 POI 不应直接导航。
   - 修复方式：把低置信度结果转成候选地点，前端展示置信度并等待确认。

### 6.5 面试官追问时的回答

问：为什么不是每个 Agent 都接 LLM？

答：

> 车载系统里不是所有模块都适合 LLM。SafetyAgent、SafetyPolicy、CarControlAgent 要保持确定性；LLM 适合放在意图兜底、路线建议、最终说明生成这些语义层。这样既展示了 LLM Agent 能力，也不会牺牲安全边界。

问：真实 API 失败怎么办？

答：

> 在线演示不做静默离线兜底。真实 API 失败就暴露失败，但前端会把技术错误转换成用户能理解的原因和建议。这样面试展示时更诚实，也更接近生产系统的可观测性。

问：这个项目怎么证明不是玩具 demo？

答：

> 我做了单元测试、离线评测、Provider smoke test 和在线验收矩阵。每次改 Agent 或 Provider 都可以运行 `scripts/run_acceptance.py` 生成验收报告，而不是靠手动点页面判断。
