# 车载 Multi-Agent 项目面试问答库

更新时间：2026-05-07

适用岗位：AI 应用工程师 / LLM 应用开发 / Agent 工程方向  
项目定位：车载 Multi-Agent 端云协同系统，覆盖本地 RAG、云端 LangGraph 编排、真实外部 Provider、车规级安全拦截、本地小模型模拟和上下文管理。

## 1. 项目总览类

### Q1：请你用 1 分钟介绍这个项目。

回答重点：

- 不是普通聊天机器人，而是车载端云协同 Agent 系统。
- 车端负责意图、安全、离线执行、数据上报。
- 云端负责用户画像、知识检索、路线规划、外部生态和 LLM 决策。
- 有真实 API、LangGraph、RAG、上下文管理和测试闭环。

可直接回答：

> 这是一个面向智能座舱场景的车载 Multi-Agent 端云协同系统。我把系统拆成车端和云端两层：车端先做本地意图识别、安全拦截和离线兜底，云端用 LangGraph 编排用户画像、知识库、地图路线、天气和充电站 Provider，最后由 LLM 生成执行说明。项目里不仅有 mock 数据，也接入了 DeepSeek、高德地图和 Open-Meteo，并补了测试、网页演示和验收脚本，所以它更像一个可演示的 AI 应用工程样板，而不是单文件 demo。

### Q2：你这个项目最核心的工程价值是什么？

回答重点：

- 展示端云边界，而不是只会调 API。
- 展示 Agent 职责拆分。
- 展示安全优先级。
- 展示本地小模型受限时的上下文管理。

可直接回答：

> 核心价值是把车载 AI 应用里的几个真实工程问题串起来了：第一是端云任务边界，哪些在车端做，哪些上云；第二是 Agent 职责拆分，避免一个大模型什么都管；第三是安全优先级，危险车控指令不能交给 LLM 自由判断；第四是本地小模型资源有限，所以我做了单 Agent 上下文窗口、摘要压缩和 prompt 预算管理。

## 2. 八大 Agent 分工类

### Q3：你为什么要拆成 8 个 Agent？

回答重点：

- 对齐课程里的八大 Agent。
- 每个 Agent 职责不同，安全等级不同。
- 避免所有任务都交给一个 LLM。

可直接回答：

> 我不是为了“数量多”而拆 Agent，而是按车载业务边界拆。车端有 LocalIntentAgent、GlobalSafetyDispatchAgent、CabinVehicleControlAgent、DataUploadAgent；云端有 GlobalTripPlanningAgent、UserProfileAgent、VectorKnowledgeAgent、ExternalEcologyAgent。这样做的好处是每个 Agent 的输入输出、风险等级、是否需要 LLM、是否能调用外部 API 都是清楚的。例如车控 Agent 不能随便调用地图，安全 Agent 不允许被 LLM 覆盖，路线规划 Agent 才会调地图。

### Q4：是不是每个 Agent 都应该接 LLM？

回答重点：

- 不应该。
- LLM 是能力，不是默认实现。
- 安全和执行类 Agent 应规则优先。

可直接回答：

> 不应该每个 Agent 都强行接 LLM。我的设计是：理解、总结、解释类任务可以用 LLM；安全判断和车控执行必须规则优先。比如 GlobalSafetyDispatchAgent 可以用本地 LLM 解释“为什么拦截”，但不能让 LLM 决定是否放行；CabinVehicleControlAgent 可以用 LLM 生成执行说明，但真正执行仍然走确定性规则。

## 3. 端云协同类

### Q5：车端和云端的边界怎么划分？

回答重点：

- 车端：低延迟、安全、离线可用。
- 云端：复杂推理、外部生态、路线规划。

可直接回答：

> 我按实时性和安全等级划分。车端负责本地意图识别、安全校验、基础车控和断网兜底，因为这些需要低延迟、可控和可离线。云端负责路线规划、用户画像、外部生态和 LLM 总结，因为这些依赖地图、天气、充电站等动态数据，也更适合复杂推理。

### Q6：为什么断网时不继续调用 DeepSeek？

回答重点：

- 业务语义上的 OFFLINE 不能依赖云端。
- DeepSeek 模拟本地模型时是工程替身，不是真离线。

可直接回答：

> 严格的断网业务链路不能依赖 DeepSeek，因为 DeepSeek API 本质还是云端服务。所以项目里把真实离线路径和“DeepSeek 模拟本地小模型”区分开：默认 `mock_local` 保证纯离线可运行；`edge_deepseek_sim` 是为了模拟车端小模型的工程行为，比如上下文预算和短输出。面试里我不会说它是真正离线部署，只会说它是本地模型 Provider 的工程模拟。

## 4. DeepSeek 模拟本地小模型类

### Q7：你为什么用 DeepSeek 模拟本地小模型？

回答重点：

- 当前没有部署真实本地模型。
- 用 Provider 抽象保留替换能力。
- 用 prompt/token 限制模拟小模型资源约束。

可直接回答：

> 因为这个项目重点是 AI 应用工程，不是训练或部署模型。为了先把端云分层和本地小模型约束做出来，我新增了 `EdgeDeepSeekSimProvider`，用 DeepSeek 的低成本模型模拟车端小模型的推理行为，同时限制上下文长度和输出长度。以后如果换成 Ollama、LM Studio、llama.cpp 或车端量化模型，只需要替换 Provider，不需要改 Agent 主流程。

### Q8：这是不是“假本地模型”？

回答重点：

- 承认不是物理离线。
- 强调工程抽象和可替换性。

可直接回答：

> 是的，它不是物理意义上的离线模型，而是本地小模型的工程模拟。我在项目里明确区分了三层：`mock_local` 保证纯离线可跑；`edge_deepseek_sim` 用 DeepSeek 模拟小模型约束；真实部署可以替换为 Ollama、LM Studio、llama.cpp 或量化模型。我的重点是把 Provider 边界、上下文预算、安全边界先设计好。

### Q9：本地小模型模拟具体限制了什么？

回答重点：

- context limit
- generation buffer
- max output
- 超预算压缩

可直接回答：

> 目前配置了 `LOCAL_LLM_MAX_CONTEXT_TOKENS=7500`、`LOCAL_LLM_GENERATION_BUFFER_TOKENS=500` 和 `LOCAL_LLM_MAX_OUTPUT_TOKENS=64`。系统会计算 prompt 预算，如果上下文过长，会只保留当前输入、摘要尾部、最近两轮交互、用户偏好、车况和少量 RAG 结果。这是在模拟车端小模型上下文窗口有限、输出要短、不能承担复杂云端推理的特点。

## 5. 上下文管理类

### Q10：为什么只给本地 LLM 做上下文管理？

回答重点：

- 云端按请求接收结构化上下文。
- 本地小模型资源有限。
- 避免全局历史污染安全链路。

可直接回答：

> 因为云端 LLM 每次可以接收结构化上下文，比如用户画像、RAG、地图和天气结果，不需要维护一段不断膨胀的会话历史。真正需要上下文管理的是车端小模型，它受限于算力、窗口和延迟，所以我只对本地 Agent 做短期窗口、摘要压缩和 token 预算。这样也能避免不同 Agent 的历史互相污染。

### Q11：上下文是全局共享的吗？

回答重点：

- 不是。
- 按 `agent_id + user_id + session_id` 隔离。

可直接回答：

> 不是全局共享。我用 `agent_id + user_id + session_id` 做隔离。比如 LocalIntentAgent 的历史不会自动进入安全 Agent 或云端 Agent。这样做更贴近真实 Multi-Agent 系统，因为不同 Agent 的职责、安全等级和上下文需求不一样。

### Q12：上下文压缩怎么做？

回答重点：

- 最近 N 轮保留完整。
- 旧轮次变摘要。
- 摘要长度受限。

可直接回答：

> 当前实现是确定性压缩：最近几轮完整保留，超过窗口的旧 turn 会被转成短摘要，合并到 summary 里，并限制 summary 字符长度。真实车端上可以把摘要函数替换成本地小模型摘要，但接口边界不需要变化。

## 6. RAG 与 Provider 类

### Q13：你的 RAG 用在哪里？

回答重点：

- 本地意图识别。
- 用户画像召回。
- 云端路线知识召回。

可直接回答：

> RAG 分三类：本地意图识别会召回指令和知识规则；UserProfileAgent 会召回用户偏好；VectorKnowledgeAgent 会召回云端业务知识，比如低电量补能、长途高速优先、断网离线导航等。这样 LLM 不是凭空生成，而是基于结构化召回结果做总结。

### Q14：真实 API 接入了哪些？

回答重点：

- DeepSeek
- 高德地图
- Open-Meteo
- 高德 POI 充电站

可直接回答：

> 目前接了 DeepSeek 做 LLM，接了高德地图做地理编码、路线规划和 POI 充电站查询，接了 Open-Meteo 做天气。OpenChargeMap 和百度地图保留了 Provider 结构，但当前主要用高德替代国内地图和充电站能力。

### Q15：为什么不让车控指令调用地图？

回答重点：

- 意图分流。
- 避免无关 Provider 调用。
- 降低成本和错误。

可直接回答：

> 因为车控和路线规划是不同业务域。比如“温度调到24度”只需要座舱控制和用户偏好，不应该调地图。项目里通过 CommandType 和 LangGraph 条件边控制路由，只有 NAVIGATION 和 CHARGE_PLAN 才会进入路线规划节点。

## 7. LangGraph 类

### Q16：为什么用 LangGraph？

回答重点：

- 多 Agent 编排适合图。
- 条件路由清晰。
- 可观测路径。

可直接回答：

> 这个项目的云端流程天然适合图结构：用户画像、知识召回、生态数据、路线规划、决策总结之间有顺序，也有条件分支。LangGraph 可以把这些节点显式化，前端还能展示 graph path。相比手写 if-else，它更容易解释多 Agent 编排逻辑。

### Q17：如果 LangGraph 没安装怎么办？

回答重点：

- 默认启用 LangGraph。
- 没有时 fallback 到 lightweight graph。

可直接回答：

> 默认是启用 LangGraph。如果环境里没有安装，会回退到项目内的 lightweight graph，保证 demo 可跑。但验收时如果安装了 LangGraph，会显示 backend 是 `StateGraph`，这说明真实 LangGraph 路径生效。

## 8. 安全类

### Q18：危险指令怎么处理？

回答重点：

- 车端前置拦截。
- 不进入云端执行。
- LLM 只能解释不能放行。

可直接回答：

> 危险指令在车端前置拦截，比如加速、制动、转向、关闭 AEB 这类指令都会被 SafetyAgent 和 SafetyPolicy 拦截。拦截后不会进入云端路线规划或车控执行。即使启用本地 LLM，它也只负责生成解释，不负责改变安全决策。

### Q19：为什么安全不能完全交给 LLM？

回答重点：

- 车规场景不能把确定性安全交给概率模型。
- LLM 可用于解释，不用于裁决。

可直接回答：

> 因为车载安全是高风险场景，不能让概率模型决定是否执行制动、转向、动力相关操作。LLM 可以帮助理解和解释，但安全边界必须由规则、策略和权限系统控制。这也是我把 GlobalSafetyDispatchAgent 放在车端前置的原因。

## 9. 测试与交付类

### Q20：你怎么证明这个项目不是只能跑一条 demo？

回答重点：

- 单元测试
- 验收脚本
- Web QA
- 输入矩阵

可直接回答：

> 我做了多层验证。单元测试覆盖 Agent、Provider、RAG、安全策略、LangGraph 和前端逻辑；验收脚本覆盖离线评估、真实 Provider smoke test 和在线输入矩阵；Web QA 会访问页面、调用 `/api/run`，并截图检查桌面和移动端布局。不是只跑一条固定指令。

### Q21：你遇到过哪些问题，怎么解决的？

回答重点：

- 地图 API 失败不能假装成功。
- 安全文本误拦截。
- 前端输入不生效。
- LangGraph 默认启用。

可直接回答：

> 过程中有几个典型问题。比如真实地图 API 对“巴黎”这类跨境目的地会失败，我没有做离线假数据兜底，而是把技术错误转成用户可理解的解释。还有一次云端结果里出现“制动距离、AEB 状态”这类安全提示，被安全规则误判为危险指令，我把安全校验改成只拦截可执行危险动作。前端也修过输入不生效和结果 Markdown 渲染问题。这个过程让我更像在做真实工程，而不是写静态 demo。

## 10. 高压追问类

### Q22：这个项目是不是 API 拼接？

回答重点：

- 承认有 API 集成。
- 强调编排、边界、安全、测试。

可直接回答：

> 它确实集成了多个 API，但不是简单 API 拼接。关键在于：我做了端云任务边界、Agent 职责拆分、LangGraph 条件编排、本地 RAG、本地上下文压缩、安全前置拦截、Provider 失败解释和验收测试。API 是能力来源，工程价值在于怎么把这些能力组织成一个可靠流程。

### Q23：如果让你继续做生产化，你下一步做什么？

回答重点：

- 真本地模型
- 权限系统
- 异步任务和缓存
- 观测与评估

可直接回答：

> 我会先把 `edge_deepseek_sim` 替换成真实本地模型，比如 Ollama 或 llama.cpp 的量化模型；然后补权限系统，把车控动作分成可执行、需确认、禁止三类；再做 Provider 缓存和超时重试；最后加线上评估和日志看板，比如意图准确率、拦截召回率、API 失败率和平均延迟。

### Q24：作为应届生，你在这个项目里体现了什么能力？

回答重点：

- 不只会写 demo。
- 有工程拆分和风险意识。
- 能把 AI 能力落到业务场景。

可直接回答：

> 我觉得这个项目体现了三个能力：第一是 AI 应用工程能力，能把 LLM、RAG、Agent、Provider、前端展示和测试串起来；第二是架构拆分能力，能区分车端、云端、安全和执行边界；第三是风险意识，没有把车载安全交给 LLM，也没有在真实 API 失败时编造结果。这些能力和 AI 应用工程师岗位是对齐的。

## 11. 30 秒收尾表达

可直接回答：

> 这个项目对我最大的价值不是“接了某个模型”，而是让我把 LLM 应用从 demo 思维推进到了工程思维：任务要拆 Agent，模型要分端云，安全要前置，RAG 要可解释，Provider 要能失败，前端要能演示，测试要能证明。作为应届生，我会把它作为我理解 AI 应用工程的核心项目来讲。
