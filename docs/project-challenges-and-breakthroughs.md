# 项目重大困难与关键突破记录

日期：2026-05-11

## 1. 从单文件 Demo 到工程化多 Agent

### 困难

项目最初只是一个单文件 offline demo，逻辑集中在一个脚本里：意图识别、安全校验、车控执行、云端调度、RAG 结果都混在一起。

这种 demo 能跑，但面试时很容易被追问：

- Agent 之间的边界在哪里？
- 端云怎么协作？
- 后续接真实 API 怎么扩展？
- 日志和测试怎么做？

### 突破

将系统拆成多层目录和多类 Agent：

- `agents/vehicle/`：车端 Agent
- `agents/cloud/`：云端 Agent
- `agents/orchestrator/`：全局调度
- `runtime/`：工具注册和 trace
- `providers/`：外部接口抽象
- `memory/`：上下文和 pending 状态
- `web_demo/`：可视化演示

### 工程价值

项目从“能跑的脚本”变成了“可扩展的系统”。这也是 AI 应用工程师和普通 demo 编写的区别：不是把功能堆在一起，而是先建立稳定的工程边界。

## 2. 八大 Agent 与真实工程类数量不一致

### 困难

最初项目计划里是八大 Agent，但随着功能完善，代码中出现了更多 Agent / 模块：

- `InputRewriteAgent`
- `DestinationConfidenceAgent`
- `EnergyPolicyAgent`
- `RouteProviderAgent`
- `CloudDecisionAgent`
- `DataUploadAgent`

这容易让人困惑：到底是不是八大 Agent？

### 突破

采用“八大业务域 + 支撑 Agent”的解释方式：

- 八大业务域用于面试讲解和产品理解。
- 支撑 Agent 用于工程边界拆分，不破坏整体业务架构。

### 工程价值

真实项目不会为了凑概念强行让一个类承担过多职责。合理拆分支撑 Agent，反而体现了工程成熟度。

## 3. 关键词命中导致误判

### 困难

早期 RAG 和意图识别依赖关键词，导致一些问题：

- “导航去北京的蔚来中心”可能只命中“蔚来中心”，忽略“北京”。
- “去上海最漂亮的地方”会被当作导航，但目的地其实不明确。
- RAG 面板显示过多召回项，让人感觉只要命中“导航”就召回。
- 用户画像、补能阈值、安全规则和文档知识混在一起，看起来像“为了 RAG 而 RAG”。

### 突破

将意图理解拆成更细的结构：

- `IntentFrame`
- `IntentRuleEngine`
- `SlotExtractor`
- 检索可靠性判断
- 本地 LLM fallback
- 目的地澄清策略
- 结构化规则库 `RuleKnowledgeAgent`
- 文档 RAG `DocumentRAGAgent`

同时对目的地引入：

- 明确目的地
- 常用地点
- 泛地点
- 城市级地点
- 低置信候选
- 多候选澄清

并对知识层重新分层：

- 安全规则、补能阈值、路线策略、舒适性规则走结构化规则库。
- 车主手册、功能说明、服务政策等长文本知识走文档 RAG。
- 用户画像由 `UserProfileAgent` 独立读取，不再混入 RAG 召回。
- 文档 RAG 数据源升级为 `rag/corpus/*.md` 文件型 corpus，补充 AEB、低电量、离线导航、天气辅助驾驶、补能站点、目的地澄清等常见车载文档。

进一步补充“RAG 按需启用”策略：

- `VectorKnowledgeAgent` 不再默认把文档 RAG 放进每一条链路。
- 普通导航、车控、补能规划只启用结构化规则，不启用 `DocumentRAGAgent`。
- `INFO_QUERY` 或包含“是什么、为什么、怎么、如何、介绍、说明、解释、含义、手册、规则、政策、注意事项”等说明型表达时，才启用 `DocumentRAGAgent`。
- 前端 `agent_trace` 只展示本次真正启用的知识 Agent，避免导航任务里看起来“明明没有问知识，却调用了文档 RAG”。
- 安全复核也做了区分：像“AEB 是什么”这种安全功能说明不能被误判为“关闭 AEB”的危险执行请求。

### 工程价值

这一步让系统从“关键词 demo”升级为“业务状态机 + 语义理解 + 知识类型分层”。面试时可以强调：AI 应用不能只靠关键词召回，尤其是导航这种高风险、高确定性要求场景；同时 RAG 也不是万能数据库，确定性规则应走规则库，长文本解释才适合走 RAG。

按需启用 RAG 的工程价值在于：减少无关召回、降低本地小模型上下文噪声、提升 trace 可信度。面试时可以明确说明：我不是为了展示 RAG 而把所有请求都接入 RAG，而是根据任务类型决定知识来源。

## 4. 模糊目的地不应该是错误

### 困难

早期当用户输入“去北京”“去高老庄”“去霓虹蔚来中心”这类模糊或低置信目的地时，系统容易返回类似接口失败的错误。

但真实产品里，这不是错误，而是正常交互状态。

### 突破

新增状态：

```text
NEEDS_CLARIFICATION
```

并引入：

- `PendingClarificationStore`
- `PendingActionStore`
- 目的地候选列表
- 用户确认后继续执行

前端也改为展示“需要确认目的地”，而不是生硬报错。

### 工程价值

这是业务建模上的重要突破：很多 AI 产品的问题不是模型答错，而是状态建模不完整。把“澄清”作为正常状态后，系统更像真实产品。

## 5. 车规安全不能交给云端 LLM

### 困难

用户质疑过：“云端给出的回答会不会再经过一次安全审查？”这暴露了一个关键问题：如果云端 LLM 生成了危险车控建议，车端是否会无条件执行？

### 突破

安全链路调整为：

```text
原始输入安全检查
  -> 重写输入安全检查
  -> 车辆状态安全策略
  -> 云端结果车端复核
  -> 执行层
```

并由 `GlobalSafetyDispatchAgent` 统一负责安全判断和云端结果复核。

### 工程价值

这体现了车载 AI 的核心原则：

**LLM 不拥有最终安全裁决权。**

面试时这点非常重要，因为车载、医疗、金融等场景都会关注高风险输出如何被拦截。

## 6. 同一句“加速到100km/h”不能一刀切

### 困难

早期危险指令直接拦截，但用户指出：如果车辆在高速上，100km/h 可能并不违法；如果在城市道路，则一定危险。

### 突破

引入车辆上下文：

- 当前道路类型 `RoadType`
- 限速 `speed_limit_kmh`
- 当前车速
- 辅助驾驶模式

安全策略开始区分：

- 高速 + 限速允许：进入 `NEEDS_DRIVER_CONFIRMATION`
- 城市 / 超限：`BLOCKED`
- AEB、制动、转向等危险控制：强拦截

### 工程价值

这一步把安全策略从“关键词拦截”升级为“上下文感知安全策略”。这是车载场景面试里很有含金量的点。

## 7. 低电量不应只在用户输入“电量低”时触发

### 困难

用户指出：低电量这种事情应该是车辆状态事件，而不是只有用户说“电量低”才出现。

### 突破

引入：

- `VehicleStateMonitorAgent`
- `EnergyPolicyAgent`
- 低电量阈值
- 严重低电量阈值
- 舒适性耗电限制

现在逻辑是：

- 电量低于 20%：导航时给补能建议。
- 电量低于 10%：导航前进入补能确认。
- 电量极低且请求座椅加热：阻止舒适性耗电功能。

### 工程价值

这让项目具备“状态驱动”的产品思维。车载系统不是纯聊天机器人，而是持续感知车辆状态并主动调整策略。

## 8. 本地小模型上下文管理

### 困难

用户明确提出：云端 LLM 可以不主动做上下文管理，但车端本地 LLM 是小参数离线模型，必须控制上下文窗口。

早期如果简单把所有历史塞进 prompt，会出现：

- token 不可控
- 旧噪声影响当前判断
- `UNKNOWN`、`BLOCKED` 等不适合进入长期记忆的内容污染上下文

### 突破

实现 `LocalAgentContextManager`：

- 以 `agent_id + user_id + session_id` 隔离上下文。
- 只保留最近窗口。
- 溢出历史压缩成摘要。
- 过滤不适合记忆的 turn。
- prompt 中加入车辆状态、用户偏好、RAG 召回和预算信息。

### 工程价值

这让本地 LLM 不只是“另一个 API 调用”，而是具备端侧模型工程约束：小上下文、短输出、低延迟、可控记忆。

## 9. 输入重写与记忆补全

### 困难

用户希望系统理解“去老地方”这类上下文相关表达。早期如果不做输入重写，后续意图识别和地图解析都会因为输入不完整而失败。

### 突破

新增 `InputRewriteAgent`：

- 使用最近导航目的地补全指代表达。
- 对“去某地”补全为“导航去某地”。
- 支持本地 LLM 输出结构化 JSON。
- 原始输入和重写输入都要经过安全检查。

### 工程价值

输入重写把用户自然语言转成更稳定的业务指令，同时保留安全边界。它让后续 Agent 更容易处理输入，也让链路更接近真实智能座舱体验。

## 10. 真实 Provider 接入后不能再用离线假数据兜底

### 困难

用户多次指出：在线演示不能失败后偷偷用离线内容兜底，否则看起来像 mock API。

例如：

- 地图路线是否真的调用？
- POI 是否真实？
- 天气是否真实？
- 失败时是否解释原因？

### 突破

引入 Provider 抽象和 Provider 状态面板：

- DeepSeek LLM
- 高德地图路线
- 高德 POI
- Open-Meteo 天气
- 离线 Provider
- Provider smoke test

在线模式下真实 Provider 失败时，系统给出智能错误解释或进入澄清，不再伪造成功。

### 工程价值

这一步提升了项目可信度。面试官能通过 trace 和 Provider 面板看到哪些是真实接口，哪些是本地模拟。

## 11. 云端工具并行化

### 困难

云端路线规划链路中有多个耗时点：

- 用户画像
- RAG 召回
- 天气
- 补能站
- 地理编码
- 地图路线
- LLM 最终说明

如果全串行，演示慢；如果乱并行，会破坏上下文依赖。

### 突破

将云端图拆成两段并行：

```text
context_parallel:
  user_profile.lookup
  knowledge.retrieve
  user_profile.route_preference

provider_parallel:
  ecology.snapshot
  route.context
```

并新增 `RouteProviderAgent`，将地理编码和地图路线工具从 `GlobalTripPlanningAgent` 内部拆出。

### 工程价值

这一步体现了对并行边界的理解：不是看到两个 Agent 就并行，而是分析数据依赖，只并行独立 IO，最后保持 LLM 单点收敛。

## 12. 前端可观测面板反复不同步

### 困难

项目多次遇到“后端改了、测试过了、但浏览器页面还是旧的”的问题。原因包括：

- 浏览器缓存静态资源。
- ES module 子依赖没有版本号。
- 旧端口进程未杀掉。
- CSS/JS/HTML 版本不同步。

### 突破

形成 `docs/frontend-sync-troubleshooting.md`：

- 入口资源加版本号。
- ES module 子模块也加版本号。
- 服务端加 no-store header。
- 重启目标端口进程。
- 用 API 和静态资源验证辅助判断。

### 工程价值

这不是算法问题，但是真实工程里很常见。能把这种问题沉淀成排障手册，说明项目不是一次性 demo，而是在持续工程化。

## 13. Agent 调用链展示从“摆样子”到“可解释”

### 困难

早期前端 Agent 调用链只是列出名字，容易被质疑：

- 这些 Agent 真的被调用了吗？
- 输出对应哪个 Agent？
- 哪些是车端，哪些是云端？
- 哪些并行，哪些串行？

### 突破

前端调整为：

- Agent 左侧展示职责、端云标签。
- 右侧展示对应工具输出。
- 工具输出和 Agent 对齐。
- 并行 Agent 显示“并行收集”。
- `RouteProviderAgent` 独立展示地图工具。
- `CloudDecisionAgent` 独立展示最终决策。

### 工程价值

这让前端从 UI 展示变成了工程可观测面板。面试中不只展示结果，还能展示系统如何得到这个结果。

## 14. 数据闭环不能乱写偏好

### 困难

数据闭环早期容易把每次输入都写入偏好，可能导致：

- 澄清态也更新偏好。
- 被拦截指令污染记忆。
- 临时意图被当成长期偏好。

### 突破

偏好更新更谨慎：

- 澄清态不更新偏好。
- 拦截态不写入语义长期记忆。
- 只在可学习事件中记录偏好变化。
- 前端数据闭环面板显示事件记录和偏好更新。

同时将长期记忆从简单计数升级为结构化画像：

```json
{
  "route_preference": {
    "value": "高速优先",
    "evidence_count": 3,
    "confidence": 0.74,
    "source": "repeated_behavior"
  }
}
```

这里 `evidence_count` 和 `confidence` 被刻意拆开：

- `evidence_count` 表示有多少次行为证据支持这个偏好，是客观计数。
- `confidence` 表示系统对该偏好结论的可信程度，由证据类型、证据权重、正负反馈和边际递减共同决定。

因此 `confidence` 不再只等于次数，而是综合：

- 用户是否显式确认，例如“以后都走高速”。
- 行为是否稳定重复。
- 场景是否一致，例如长途、高速、市区通勤要区分。
- 是否存在反例，例如用户后续说“这次不要走高速”。
- 证据是否近期发生。
- 来源是否可靠，例如显式设置高于模型推断。

当前版本已经支持正向证据、负向证据、显式偏好、行为证据和边际递减。后续仍可继续接入更细的时间衰减和场景分桶，而不需要推翻数据结构。

### 工程价值

真实 AI 产品的数据闭环不是“什么都存”，而是要判断哪些数据值得学习，哪些必须过滤。长期记忆也不能简单等同于历史记录或次数统计，而应该是带证据、置信度和来源的可解释画像。

## 15. 最终项目能力提升

经过这些困难和突破，项目从最初 demo 进化为：

- 可 offline 运行
- 可接真实 API
- 可展示端云协同
- 可按任务类型启用 RAG
- 可解释 Agent 调用链
- 可处理模糊输入
- 可执行安全拦截
- 可管理本地小模型上下文
- 可进行并行云端编排
- 可生成验收报告
- 可用于面试讲解

## 16. 面试总结话术

可以这样总结项目演进：

> 这个项目最有价值的部分不是一开始就写了很多 Agent，而是在不断发现真实产品问题后，把它工程化地修正。比如关键词召回会误判，我就拆出意图框架和目的地澄清；危险指令不能只看词，我就加入车辆状态和安全策略；低电量不能靠用户输入触发，我就加入状态监控和能源策略；云端链路慢，我没有粗暴并行 LLM，而是拆出独立 IO 工具并行；前端看不出 Agent 是否真的调用，我就把 trace 和 Agent 对齐。  
> 
> 同时我也重新设计了知识层：确定性规则走结构化规则库，车主手册和功能说明走按需文档 RAG，用户画像由独立 Agent 读取，避免所有信息都混在一个 RAG 召回里。  
> 
> 这些突破让项目从一个课程 demo 变成了一个可解释、可测试、可扩展的车载 AI 应用工程原型。

## 17. 工程化硬化优先级

### 困难

项目功能快速扩展后，最大的风险不再只是某个 Agent 是否能跑通，而是工程闭环是否稳定：

- 默认验收不能依赖真实 API key，否则外部网络或配额会让 CI 不稳定。
- 真实 Provider 又不能被长期忽略，否则面试演示前才发现 key、接口、字段或网络策略变化。
- 验收脚本会生成报告、截图、pytest 临时目录和运行态文件，如果不忽略，容易污染工作区。
- 多人并行改造时，需要把“什么必须进 CI、什么只做手动 smoke”写清楚，减少互相误判。

### 突破

工程化硬化按优先级拆成三层：

1. 默认 CI 只跑离线确定性回归：Mock Local LLM、Offline Provider、pytest、前端 JS 语法和稳定演示场景。
2. 真实 Provider smoke test 保持手动 opt-in：本机配置 `.env` 后运行 `python scripts/smoke_real_providers.py`，验证 DeepSeek、Open-Meteo 和高德链路。
3. 生成产物统一忽略：`reports/`、`.tmp/`、pytest 临时目录、coverage 和 Web Demo 日志都不进入版本库。

### 工程价值

这一步让项目从“本机能演示”进一步变成“可以持续协作”。CI 保证离线核心链路稳定，真实 Provider smoke 保证上线接口可验证，两者边界清晰后，既不会为了稳定性假装真实接口没问题，也不会让外部依赖拖垮默认回归。
