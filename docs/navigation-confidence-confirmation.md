# 导航置信度与候选确认设计

## 背景

车载导航指令不能只要地图 API 返回坐标就直接执行。地图返回结果只能证明“服务找到了一个地点”，不能证明“这个地点就是用户想去的地方”。例如“导航去世博园”可能对应多个城市或多个 POI，如果系统直接选择第一个结果，会产生错误导航风险。

## 当前策略

本项目新增 `DestinationConfidenceAgent`，放在导航进入云端路线规划前：

1. 本地意图 Agent 识别为 `NAVIGATION`。
2. 安全 Agent 先做危险指令拦截。
3. 目的地置信度 Agent 处理动态导航目的地。
4. 高德 POI Provider 召回候选地点。
5. LLM 只基于地图候选做语义置信度判断，不允许编造坐标。
6. 只有绝对高置信度才进入路线规划，否则返回 `NEEDS_CLARIFICATION`。

## 自动执行条件

以下情况允许直接执行：

- 显式 GPS，例如 `导航去121.50,31.25`。
- 内置高确定性目的地，例如 `回家`、已配置的 `蔚来中心`。
- 用户高频成功导航过的常用地点，例如某用户多次成功导航到 `世博园` 后，可进入个人常用地点白名单。
- 项目内配置的本地高确定性地标，例如 `外滩`、`陆家嘴`、`东方明珠`。
- 在无在线候选 Provider 的单元测试环境里，强特异性目的地可继续交给后续执行层，例如 `北京东方广场蔚来中心`。

## 必须澄清条件

以下情况不会直接导航：

- 泛化地点：`世博园`、`机场`、`车站`、`公园` 等。
- 连锁门店但缺少城市、商圈或门店定位：`北京蔚来中心`、`蔚来中心` 的动态变体。
- 地图候选多个，LLM 判断置信度不足。
- 地图候选与用户输入不一致或低置信度。

## 前端交互

`NEEDS_CLARIFICATION` 是正常业务状态，不是 502 错误。前端会展示：

- 澄清问题
- 建议补充方式
- 地图候选地点列表
- 候选名称、地址、置信度、来源

用户点击候选后，前端使用候选 GPS 发起新导航请求，表示“用户已确认该具体目的地”，避免再次被泛化名称拦截。

## 面试表达重点

可以这样解释：

> 我把目的地处理拆成两层置信度：地图匹配置信度和用户意图唯一性置信度。地图 API 负责召回候选，不直接决定执行；LLM 负责在候选集合上判断是否足够明确。对于车载场景，导航误执行有安全和体验风险，所以默认策略是保守的：除 GPS、用户常用地点、明确配置地点和绝对高置信度地点外，其他情况都进入候选确认。
