# 知识层分层升级：规则库与文档 RAG

本文记录本次知识层升级的设计原因和实现结果。核心变化是：不再把所有知识都笼统称为 RAG，而是区分结构化规则库和非结构化文档 RAG。

## 为什么要升级

早期项目中 `VectorKnowledgeAgent` 同时承担了用户画像、路线规则、补能策略和 RAG 召回展示，容易造成两个问题：

- 确定性规则被包装成 RAG，看起来像“为了 RAG 而 RAG”。
- 用户画像、路线规则、长文本知识混在同一个召回面板里，面试时很容易被追问“这些内容普通数据库不就够了吗”。

真实车载系统里，安全阈值、补能阈值、用户画像、车辆状态、常用地点这类信息应该走数据库、配置表或规则引擎，而不是依赖向量相似度。RAG 更适合车主手册、功能说明、服务政策这类长文本知识。

## 新的知识层划分

### 结构化规则库

由 `RuleKnowledgeAgent` 负责。

适用范围：

- 电量阈值与补能策略。
- 长途路线策略。
- 舒适温度范围。
- 离线导航兜底规则。
- 确定性业务规则。

特点：

- 确定性强。
- 可解释、可审计。
- 不依赖 LLM 猜测。
- 不适合用向量召回做最终决策。

### 文档 RAG

由 `DocumentRAGAgent` 负责。

适用范围：

- 车主手册问答。
- AEB、座椅加热等功能说明。
- 换电服务政策。
- 维修保养、售后规则等长文本知识。

特点：

- 用户问法多样。
- 原文较长。
- 需要检索片段后由 LLM 解释。
- 适合做 RAG。

### 兼容入口

保留 `VectorKnowledgeAgent` 作为兼容入口，但它现在更像 `HybridKnowledgeAgent`：

- 导航、补能、车控：优先走 `RuleKnowledgeAgent`。
- 信息查询：走 `DocumentRAGAgent`。
- 用户画像：不再混入知识召回，由 `UserProfileAgent` 单独负责。

这样保持了原有云端编排节点 `knowledge.retrieve` 的稳定性，同时让内部职责更合理。

## RAG 按需启用策略

本次继续补充了“按需启用 RAG”的机制：`knowledge.retrieve` 仍然是云端编排中的统一知识入口，但入口内部会先判断本次任务是否真的需要文档 RAG。

启用规则如下：

- `NAVIGATION`、`CHARGE_PLAN`、`CAR_CONTROL`：默认只查结构化规则库，不进入文档 RAG。
- `INFO_QUERY`：启用文档 RAG，用于回答 AEB、座椅加热、低电量策略、离线导航等说明类问题。
- 用户输入包含“是什么、为什么、怎么、如何、介绍、说明、解释、含义、手册、规则、政策、注意事项”等说明型表达时，即使意图尚不明确，也允许启用文档 RAG。
- 普通导航指令，例如“导航去蔚来中心”，不会因为命中“导航”关键词就召回长文本手册。

这样做的原因是：车载系统里的规则、阈值和状态决策应保持确定性；文档 RAG 只在需要解释长文本知识时才进入链路，避免无关召回污染 LLM 输入，也避免前端展示“看起来什么都用了”的虚假复杂度。

前端观测面板也同步调整为“实际启用才展示”：

- 普通导航链路显示 `RuleKnowledgeAgent`，不显示 `DocumentRAGAgent`。
- `AEB是什么` 这类说明问题显示 `DocumentRAGAgent`。
- 如果本次没有文档 RAG 召回，`RAG 召回知识` 面板不会为了凑数量展示无关内容。

## 当前数据来源

结构化规则库来自：

- 课程作业内置业务规则。
- 项目开发过程中沉淀的安全、补能、导航和舒适性规则。
- `rag/documents.py` 中标注为 `knowledge_type = structured_rule` 的条目。

文档 RAG 数据来自：

- `rag/corpus/*.md` 下的 Markdown 文档文件。
- 当前内置了 AEB、座椅加热、低电量补能、离线导航、天气与辅助驾驶、补能站点、目的地澄清、数据闭环与隐私边界等常见车载 AI 场景文档。
- `rag/documents.py` 中标注为 `knowledge_type = document_rag` 的条目保留为兼容兜底，正常情况下优先加载文件型 corpus。

当前仍是离线可运行的轻量实现。真实项目中可以扩展为：

- 规则库：数据库、配置中心、规则引擎。
- 文档 RAG：车主手册、售后知识库、服务政策、企业内部知识库，配合 FAISS/Milvus 等向量库。

## 当前文档 Corpus

当前文件型文档库位于 `rag/corpus/`：

- `vehicle_manual_aeb.md`：AEB 自动紧急制动系统说明。
- `seat_heat_usage.md`：座椅加热与低电量舒适性策略。
- `energy_policy_low_battery.md`：低电量、严重低电量与补能策略。
- `navigation_offline_fallback.md`：离线导航与断网兜底。
- `assisted_driving_weather.md`：天气对辅助驾驶的影响。
- `charging_station_usage.md`：补能站点使用与 POI 结果解释。
- `navigation_destination_clarification.md`：模糊目的地澄清策略。
- `privacy_data_loop.md`：数据闭环、长期记忆与隐私边界。

每个文档通过文件头声明 `doc_id`、`topic` 和 `keywords`，由 `DocumentRAGAgent` 加载为检索文档。这样面试时可以明确说明：RAG 的数据源不是硬编码空壳，而是项目内可扩展的文件型知识库。

## 面试表达

可以这样讲：

> 我后期没有继续把所有知识都叫 RAG，而是把知识层拆成了结构化规则库和文档 RAG。安全阈值、补能策略、用户画像这类确定性信息走规则库或数据库；车主手册、功能说明、服务政策这类长文本知识才走 RAG。这样既避免了“为了 RAG 而 RAG”，也让系统更符合车载场景对确定性、可解释和安全审计的要求。

这个升级点体现的是：RAG 不是万能数据库，AI 应用工程需要根据知识类型选择合适的存储和检索方式。
