# 面试讲解指南

## 1. 项目一句话介绍

这是一个车载智能座舱端云协同 Multi-Agent 原型系统，支持本地意图识别、云端多 Agent 调度、RAG 知识增强、安全拦截和断网兜底。

## 2. 简历描述

设计并实现车载端云协同 Multi-Agent 原型系统，拆分 SafetyAgent、LocalIntentAgent、CarControlAgent、NavAgent、CloudScheduleAgent 等 8 个模块；通过本地知识库模拟 RAG 意图识别、用户画像召回和路线规划；实现危险车控指令拦截、断网本地兜底、统一消息协议和核心单元测试。

## 3. 项目亮点

- 不是普通聊天机器人，而是带执行链路的 AI 应用系统。
- 有明确业务场景：导航、车控、补能、用户偏好。
- 有端云协同设计：在线走云端，离线走本地。
- 有车规安全意识：危险指令本地优先拦截。
- 有 RAG 思想：本地知识库、用户画像、路线知识召回。
- 有工程质量：模块化目录、统一协议、自动化测试、文档。

## 4. 面试讲解顺序

1. 先讲业务场景：智能座舱中的导航、车控、充电规划、偏好推荐。
2. 再讲架构：车载执行层、端云通信层、云端决策层。
3. 再讲 Agent：车端 4 个 Agent，云端 4 个 Agent。
4. 再讲关键链路：输入、意图、安全、网络判断、执行。
5. 最后讲工程取舍：为什么先 offline，为什么模拟 RAG，后续如何扩展。

## 5. 高频问答

### 问：为什么没有直接接入大模型？

答：第一版定位是 offline 工程原型，重点验证端云协同、Agent 边界、安全拦截和 RAG 位置。模型和向量库可以作为后续替换项接入，不影响主流程。

### 问：这个项目和普通聊天机器人有什么区别？

答：普通聊天机器人主要关注回答文本，这个项目关注业务执行链路，包括意图识别、安全校验、车控执行、断网兜底和云端调度，更接近 AI 应用工程岗位的真实工作。

### 问：为什么要有统一 Message？

答：多 Agent 系统如果没有统一协议，模块之间会变成临时字符串传参，后续难以测试和扩展。`Message` 把请求 ID、用户 ID、意图、安全等级、内容和网络状态统一封装，方便端云通信和链路追踪。

### 问：为什么危险指令仍识别成 CAR_CONTROL？

答：从语义上看，加速、制动、转向属于车控意图；但它们的安全等级是 `DANGEROUS`。系统先识别意图，再由 SafetyAgent 拦截执行，这样输出中可以同时体现“它是什么”和“为什么不能执行”。

### 问：后续如何接入真实 RAG？

答：可以把当前字典和列表知识库替换为 embedding 检索。第一步可用 TF-IDF 或 BM25，第二步接 FAISS，第三步接 Milvus 或云端向量检索服务。Agent 的上层接口可以保持不变。
